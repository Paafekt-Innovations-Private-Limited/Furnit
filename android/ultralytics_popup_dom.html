For implementing Lanczos resampling on Android for a 3D reconstruction pipeline like SHARP, here is the recommended approach:

1. Implementation Recommendation

For production use, OpenCV Imgproc.resize with INTER_LANCZOS4 is the most robust choice.

  • Why: Kotlin-based custom kernels are too slow for high-resolution (2000px) bitmaps due to JVM overhead. Since RenderScript is deprecated, OpenCV provides highly optimized native (C++/NEON) instructions that outperform standard Java/Kotlin loops.
  • Alternative: If you want to avoid OpenCV's library size, use Glide or Coil with a custom BitmapTransformation, though these often wrap standard Canvas scaling (Bilinear/Bicubic) rather than true Lanczos.
// OpenCV Implementation
val targetSize = Size(1536.0, 1536.0)
Imgproc.resize(srcMat, dstMat, targetSize, 0.0, 0.0, Imgproc.INTER_LANCZOS4)

2. Kernel Size & Performance

For a 2000px to 1536px downscale (a small ratio of ~1.3x):

  • Lanczos3 (Kernel 6x6): Recommended for the best quality-to-speed tradeoff. It strikes a balance by providing sharper edges and fewer aliasing artifacts than Bilinear without the heavy computational cost of Lanczos4 (8x8) or higher.
  • Performance Tip: Since the downscale ratio is small, the visual difference between Lanczos and Bicubic Interpolation might be negligible. Test Bicubic if Lanczos3 exceeds your latency budget.

When switching from Bilinear to Lanczos for quantized model inputs:
