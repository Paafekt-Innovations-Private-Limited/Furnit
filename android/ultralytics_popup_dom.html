We have an Android app with two flows:

1) SharpRoomActivity: shows a 3D Gaussian splat room (PLY) in a WebView using Three.js + SparkJS SplatMesh. User sees the room and can orbit/zoom.

2) Brain icon: user taps a "brain" button which starts FurnitureFitActivity with the same room (ROOMFOLDER with room.ply). FurnitureFitFragment runs "SmartyPants" — AI furniture detection. The fragment gets ROOMID, ROOM_FOLDER, room dimensions, and photo orientation from the intent.

Problem: When the user taps the brain icon, the new Activity shows a full grey screen with only a progress bar and text ("Detecting furniture...", "Preprocessing..."). The 3D room is not visible until much later (or never). The user wants: the same 3D room to remain visible from the first frame, with only the progress bar overlaid on top (semi-transparent strip at top). So: room stays as-is, progress bar overlays it.

Current implementation (FurnitureFitFragment):

  • When hasRoomBackground is true (opened from brain with a room), we set previewView.visibility = GONE (no camera feed).
  • roomSceneView (GLB) and roomPlyWebView (WebView for PLY splat) are created with visibility = GONE.
  • loadRoom3D() is called in onCreateView; it runs async (lifecycleScope.launch) and loads room.ply into roomPlyWebView via loadPlyBackground(), then sets roomPlyWebView.visibility = VISIBLE when done. For GLB it sets roomSceneView.visibility = VISIBLE.
  • So until loadRoom3D() completes, the user sees: no preview, no room view (both GONE), so only the root FrameLayout background (grey) and the progress container (semi-transparent black with progress bar) on top.
  • We already tried making roomPlyWebView.visibility = VISIBLE when hasRoomBackground at view creation time, so the WebView layer is visible from the start — but the room still does not show or stays grey.

Possible causes we need help with:(1) Is the room WebView actually receiving and rendering the same PLY (same HTML/SparkJS) as SharpRoomActivity? loadPlyBackground() copies room.ply to internal storage and builds HTML with buildPlyViewerHtml(); could the WebView be loading slowly or failing silently so the user sees grey for a long time?(2) Should we not start a new Activity at all — and instead show the progress overlay (progress bar + "Detecting furniture...") as a dialog or overlay View on top of SharpRoomActivity's WebView so the room never disappears? If so, how to structure: run detection in background, show overlay in same Activity, then transition to detection results without replacing the room?(3) Or should we pre-load the room in FurnitureFitActivity before the fragment is shown (e.g. in Activity onCreate load the PLY into a WebView and pass it or show it before Fragment attaches) so the room is already visible when the user sees the new screen?(4) Any other Android/WebView best practice to "keep the previous screen content visible" when opening a new Activity that needs to show progress — e.g. shared element transition with the WebView, or a single Activity with Fragment swap where the room view is kept and only the overlay is added?

We are not using YOLO; this is 3D Gaussian splatting room viewer plus a separate furniture-detection flow that should overlay progress on the same room.

Ultralytics AI

To ensure the 3D room remains visible during furniture detection, you should avoid the overhead of a new Activity. Loading Gaussian Splatting data in a new WebView triggers a full re-initialization of the rendering engine, causing the "grey screen" lag.

Recommended Strategy

  • Single Activity Architecture: Do not start FurnitureFitActivity. Instead, keep SharpRoomActivity active and add the progress UI as an overlay View or Fragment within the same layout. This prevents the WebView from being destroyed and recreated.
  • Layered UI: Place your progress bar in a FrameLayout or ConstraintLayout with a higher z-index (elevation) than the WebView. Set its initial visibility to GONE and toggle it to VISIBLE when the "brain" button is tapped.
  • Background Processing: Run your "SmartyPants" detection logic using a ViewModel and lifecycleScope. This allows the UI thread to keep the real-time rendering smooth while the AI processes data in the background.
<!-- activity_sharp_room.xml -->
<FrameLayout ...>
    <WebView android:id="@+id/roomPlyWebView" ... />
    
    <!-- Progress Overlay -->
    <LinearLayout 
        android:id="@+id/progressOverlay"
        android:visibility="gone"
        android:background="#80000000" 
        ...>
        <ProgressBar ... />
        <TextView android:text="Detecting furniture..." ... />
    </LinearLayout>
</FrameLayout>
